## 1. Introduction

### 1.1 Purpose

Labyrinth Legends produces documentation, visual assets, and implementation through **human expertise amplified by AI capability**. Without an operating model, acceleration becomes drift — style forks, authority confusion, and unreviewed output entering the repository.

**AB5 defines the production specification for collaborative AI-assisted production** — how humans and AI participate at every stage, who owns decisions, how documentation provides context, and how quality emerges through governance rather than hope.

AB5 answers:

> **How do humans and AI collaborate without surrendering creative ownership, documentation authority, or review discipline?**

### 1.2 Authority and Scope

| AB5 owns | AB5 does not own |
|----------|------------------|
| Collaborative **production system** responsibilities and operating specs | Gameplay rules, puzzle logic, movement |
| Human vs AI responsibility boundaries | Visual language meaning (LLDL) or asset design (AB2–AB4) |
| Documentation-first context inheritance model | Marketing strategy, store copy, campaigns (AB4 content scope) |
| Review and iteration philosophy in collaborative work | Formal gate metrics, outsourcing contracts (AB6) |
| Tool- and model-independent workflow patterns | Prompt collections, model comparisons, vendor tutorials |
| Future-proof production evolution principles | Flutter implementation, version control procedures |
| Participant roles and decision authority | Business processes, budgets, publishing schedules |

### 1.3 Inheritance

```text
Vision.md     — why the game exists
    ↓
Gameplay.md   — how the game works
    ↓
LLDL.md       — how the game looks, feels, moves, and communicates visually
    ↓
AB0 / AB1     — production philosophy and universal standards
    ↓
AB2           — playable-world asset systems
    ↓
AB3           — interface and in-product brand experiences
    ↓
AB4           — public brand communication systems
    ↓
AB5 (this doc)— collaborative AI-assisted production operating model
    ↓
AB6           — review gates and asset lifecycle
    ↓
Asset_Bible.md — integrated production handbook
```

#### Vision

[Vision](../00_Project/Vision.md) demands premium craft and player respect. Collaborative production must **accelerate quality** — not substitute shortcuts that erode trust.

#### Gameplay

[Gameplay](../01_Game_Design/Gameplay/Gameplay.md) defines mechanical truth. AI must not invent rules, reinterpret precedence, or ship gameplay logic without human-authored specification and test validation.

#### LLDL

[LLDL](../02_Design_System/LLDL/LLDL.md) is visual authority. AI generates **within** LLDL boundaries established by humans — it does not set art direction.

#### AB0 and AB1

[AB0 §5 — AI Philosophy](AB0_Asset_Philosophy_Production_Principles.md#5-ai-philosophy) establishes beliefs: AI accelerates under direction, never replaces approval. [AB1](AB1_Production_Standards.md) establishes governance culture and state model. AB5 **operationalizes** those beliefs into collaborative systems.

#### AB2, AB3, and AB4

Domain workshops define **what** to produce — game systems, interface experiences, public communication families. AB5 defines **how** humans and AI collaborate when producing work within those domains — without redefining domain specifications.

#### Asset Bible and Technical Documentation

`Asset_Bible.md` integrates all workshops. `docs/04_Technical/*` and `docs/05_AI/*` may contain implementation-adjacent guidance — AB5 remains the **Asset Bible authority** for collaborative production operating model. Technical docs must align with AB5; they do not override it on collaboration philosophy.

### 1.4 What AB5 Deliberately Does Not Own

AB5 excludes prompt libraries, specific model recommendations, Cursor/ChatGPT manuals, Flutter coding standards, git workflows, business planning, and formal QA gate definitions. If content defines *what a hazard looks like* — AB2. If it defines *how a store screenshot is composed* — AB4. If it defines *gate metrics and outsourcing SLAs* — AB6.

## 2. Universal Collaborative Production Principles

Every collaborative production system inherits these principles **once**. Individual sections apply them; they do not restate them.

### 2.1 Human Ownership

Humans retain **creative and approval ownership**. AI proposes; humans dispose. No output ships because a model rendered it successfully.

### 2.2 AI Collaboration

AI extends capability — exploration, variation, drafting, cleanup, documentation assistance — within boundaries humans define. Collaboration is structured, not ad hoc.

### 2.3 Creative Accountability

A **named human** is accountable for every deliverable class. Shared tools do not create shared ambiguity about who approved meaning and quality.

### 2.4 Documentation-First Production

Authoritative documentation is the **primary context** for both humans and AI. Undocumented work is not yet studio knowledge — per [AB0 §4.6](AB0_Asset_Philosophy_Production_Principles.md#46-documentation).

### 2.5 Authority Hierarchy

When sources conflict, **higher documentation wins** — Vision → Gameplay → LLDL → Asset Bible domain workshops → implementation. AI must not silently resolve conflicts.

### 2.6 Context Inheritance

Each production task inherits context from locked authorities and relevant domain specs (AB2–AB4). Context is **curated** — not everything is loaded; what is loaded must be truthful.

### 2.7 Review Philosophy

Review is **mandatory**, continuous, and reference-based — citing principles and authorities, not personal taste alone. AI output receives the same review rigor as handmade work — per [AB0 §5.6](AB0_Asset_Philosophy_Production_Principles.md#56-same-review-gates).

### 2.8 Quality Gates

Work advances through **explicit states** — per [AB1 §3](AB1_Production_Standards.md#3-asset-governance). AB5 defines collaborative participation at gates; AB6 defines formal gate detail.

### 2.9 Iterative Improvement

Iteration is expected. Collaborative loops return to generation only through documented feedback — not endless unguided regeneration.

### 2.10 Future-Proof Production

Systems are defined **independently of current vendors and models** — so tooling may change without rewriting studio production law.

### Collaborative Production Validation

Before any system is complete:

1. Improves **collaborative production** — not just individual speed
2. **Human ownership** remains explicit
3. **AI responsibilities** remain bounded
4. **Documentation** remains authoritative
5. **Review** remains mandatory
6. Workflow is **tool-independent**
7. System remains **scalable** for future production technologies

## 3. Collaborative Production Lifecycle

All collaborative production systems support a single lifecycle arc:

```text
Idea
    ↓
Research
    ↓
Architecture
    ↓
Planning
    ↓
Generation
    ↓
Review
    ↓
Revision
    ↓
Approval
    ↓
Repository Integration
    ↓
Maintenance
```

| Stage | Human role (summary) | AI role (summary) |
|-------|----------------------|-------------------|
| **Idea** | Frame intent, scope, authority | Assist brainstorming — no authority |
| **Research** | Select truth sources, validate relevance | Summarize, cross-reference docs — human verifies |
| **Architecture** | Decide structure, ownership, dependencies | Draft options — human selects |
| **Planning** | Define brief, acceptance criteria, domain alignment | Assist task breakdown — human approves plan |
| **Generation** | Direct, constrain, select references | Produce drafts, variations, code, imagery |
| **Review** | Judge against authorities, record findings | Assist checklist application — no self-approval |
| **Revision** | Prioritize fixes, update brief if needed | Implement directed changes |
| **Approval** | Named authority attests fit for purpose | None — approval is human-only |
| **Repository Integration** | Merge, trace, document handoff | Assist formatting, boilerplate — human commits judgment |
| **Maintenance** | Monitor drift, schedule refresh | Assist diff analysis, migration drafts — human decides |

Stages are **iterative**, not strictly linear. Maintenance may return to Research when systems evolve — always through the same collaborative discipline.

### Collaborative Production System Template

Every system in §4–§8 follows this internal structure:

| Section | Purpose |
|---------|---------|
| **Purpose** | Why this collaborative production system exists |
| **Participants** | Who participates — role types, not vendor products |
| **Responsibilities** | Human, AI, and shared ownership |
| **Workflow** | How work moves through collaboration |
| **Quality Expectations** | Validation, review loops, documentation usage |
| **Authority** | Truth sources, conflict resolution, approval |
| **Consistency Rules** | Alignment across people, AI, and time |
| **Future Evolution** | Scalability as tools and technologies change |

## 4. Collaborative Philosophy

**Primary responsibility:** Establish the enduring partnership model between human expertise and AI capability.

### Purpose

Collaborative Philosophy anchors **why** the studio uses AI — and the non-negotiable beliefs that prevent automation from becoming authority drift.

### Participants

| Participant | Role |
|-------------|------|
| **Human — Creative Authority** | Owns intent, taste boundaries, final approval |
| **Human — Production Lead** | Owns scope, schedule, and review routing |
| **Language Model** | Drafts, explores, summarizes within directed context |
| **Generative Media AI** | Produces visual/audio variations within reference locks |
| **Code Generation AI** | Drafts implementation within architecture and test requirements |
| **Reviewer** | Validates against authorities — may be human or human-directed review role |
| **Approval Authority** | Human with explicit gate authority — typically Owner or delegate |

### Responsibilities

| Owner | Responsibility |
|-------|----------------|
| **Humans** | Define what the game is; set boundaries; approve all shipped work |
| **AI** | Accelerate execution inside boundaries; surface options; never self-approve |
| **Shared** | Iteration loops — human directs, AI responds, human validates |
| **Decision authority** | Humans exclusively — escalation to [Decisions](../00_Project/Decisions.md) when authorities conflict |

### Workflow

1. **Intent first** — no generation without documented purpose and authority check
2. **Boundaries before prompts** — LLDL and domain specs (AB2–AB4) constrain all creative AI
3. **Acceleration, not abdication** — AI shortens cycles; humans shorten mistakes
4. **Same moral weight** — AI-assisted output is not "lower tier" for review — per AB0
5. **Ethical production** — no laundering unlicensed sources; per [AB0 §5.8](AB0_Asset_Philosophy_Production_Principles.md#58-ethical-production)

### Quality Expectations

- Every collaborative session produces **traceable rationale** — what was asked, what was accepted, what was rejected
- Philosophy violations (AI as authority, skipped review, undocumented context) are **stop conditions**
- Quality is measured by **alignment with authorities**, not by generation speed

### Authority

| Source | Role in philosophy |
|--------|---------------------|
| [AB0 §5](AB0_Asset_Philosophy_Production_Principles.md#5-ai-philosophy) | Philosophical foundation — AB5 does not redefine |
| [Vision](../00_Project/Vision.md) | Product intent ceiling |
| [LLDL](../02_Design_System/LLDL/LLDL.md) | Visual meaning ceiling |
| **AB5 Collaborative Philosophy** | Operating beliefs for all collaborative systems |

Conflict resolution: preserve higher authority → report in review package → do not ship until resolved.

### Consistency Rules

- All domain production (AB2–AB4) invokes the **same partnership beliefs**
- No team or contributor may adopt "AI exception" culture — faster output, same gates
- Prompt direction encodes **approved standards in language** — per AB0; prompt libraries are AB6/technical detail, not AB5 philosophy

### Future Evolution

- New model capabilities do not change ownership — only **throughput and option richness**
- Philosophy is revised rarely — via documented workshop with Human approval
- Tool vendors may change; **human ownership and documentation authority** do not

## 5. Collaborative Roles

**Primary responsibility:** Define who does what — eliminating ambiguous handoffs between humans, AI, and review functions.

### Purpose

Collaborative Roles translate philosophy into **assignable production responsibility** — so every task has a human owner, bounded AI participation, and a clear review path.

### Participants

| Participant | Role |
|-------------|------|
| **Specification Author** | Human — writes or curates authoritative docs AI inherits |
| **Domain Steward** | Human — owns AB2, AB3, or AB4 alignment for a workstream |
| **Implementer** | Human — may use code generation AI under architecture constraints |
| **Asset Author** | Human — may use generative media AI under LLDL and domain locks |
| **AI Operator** | Human — runs AI tools; does not equal "AI" as participant |
| **Engineering Reviewer** | Human — Codex role; architecture and maintainability |
| **Product Reviewer** | Human — ChatGPT role; product, UX, LLDL compliance |
| **Approval Authority** | Human — final gate per AB1 state model |

### Responsibilities

| Role | Human responsibilities | AI may assist | AI must not |
|------|------------------------|---------------|-------------|
| **Specification Author** | Maintain LLDS truth | Draft sections for human edit | Publish undocumented design law |
| **Domain Steward** | Enforce AB2/AB3/AB4 boundaries | Summarize domain specs | Redefine domain ownership |
| **Implementer** | Tests, architecture compliance | Code drafts, refactors | Merge unreviewed gameplay logic |
| **Asset Author** | Reference locks, final craft | Variations, cleanup | Set art direction |
| **AI Operator** | Tool configuration within AB5 | N/A | Approve output |
| **Reviewers** | Record verdicts in review packages | Checklist assistance | Replace human judgment |
| **Approval Authority** | Explicit approval attestation | None | Delegate to model |

### Workflow

1. **Task assignment** — every work item names a human owner before AI use
2. **Role declaration** — participants identify which hats they wear (author, operator, reviewer)
3. **Handoff rules** — AI output always returns to a **human role** — never to another unattended AI chain without review
4. **Escalation** — ambiguity routes to Domain Steward → Production Lead → Owner

### Quality Expectations

- Role map is **documented per major task** in review packages
- No "anonymous AI contribution" in shipped work
- Cross-discipline tasks name **multiple human roles** with a single approval authority

### Authority

- Role definitions in AB5; **gate authority** detail in AB1 and AB6
- [AGENTS.md](../../AGENTS.md) defines agent **roles** (Cursor, Codex, ChatGPT, Human) — AB5 aligns with but does not replace AGENTS governance

### Consistency Rules

- Same role names across documentation, review packages, and handoffs
- Vendors and freelancers receive **role briefs** — not open-ended AI access
- Solo studio operation still maintains **separation of author and reviewer** when possible

### Future Evolution

- New roles (e.g., localization lead) **extend** the map — do not fragment accountability
- Automation may assume **operator** tasks — never **approval authority**

## 6. Collaborative Workflow

**Primary responsibility:** Describe how work moves from intent to integrated deliverable through human–AI collaboration.

### Purpose

Collaborative Workflow is the **operational spine** — connecting lifecycle stages to daily production behavior without binding to a specific IDE or vendor.

### Participants

| Participant | Typical involvement by stage |
|-------------|-------------------------------|
| **Human — Owner / Lead** | Architecture, Planning, Approval |
| **Human — Author** | Research, Generation, Revision |
| **Language Model** | Research, Planning drafts, Generation, Revision |
| **Generative Media AI** | Generation, Revision (visual/audio) |
| **Code Generation AI** | Generation, Revision (code/docs) |
| **Reviewer** | Review |
| **Approval Authority** | Approval |

### Responsibilities

| Stage | Human owns | AI assists |
|-------|------------|------------|
| **Research** | Source selection, truth verification | Summarization, gap identification |
| **Architecture** | Structure, dependencies, scope class (MVP/Post-MVP) | Option drafting |
| **Planning** | Brief, acceptance criteria, required reading list | Task decomposition drafts |
| **Generation** | Constraints, references, iteration direction | Output production |
| **Review** | Verdict, blocking issues, recommendations | Applying checklists to drafts |
| **Approval** | Attestation | None |
| **Integration** | Repository judgment, doc updates, review package | Boilerplate, formatting |
| **Maintenance** | Drift detection, refresh triggers | Impact analysis drafts |

### Workflow

```text
Brief (human) + Required Reading (LLDS)
        ↓
Context Package (curated docs — not whole repo dump)
        ↓
Generation (AI within boundaries)
        ↓
Self-check (human operator — not AI self-approval)
        ↓
Review (Codex / ChatGPT / peer per task class)
        ↓
Revision loop (directed — max cycles without re-brief)
        ↓
Approval (named human)
        ↓
Integration (code/docs/assets + review package if major)
        ↓
Maintenance (scheduled or triggered refresh)
```

**Documentation tasks** follow LLDS compiler workflow — per [Asset Bible README](README.md#7-documentation-methodology).

**Asset tasks** validate in **domain context** — maze board, screen, or store frame — not isolated canvas.

**Major tasks** require review packages — per [AGENTS.md](../../AGENTS.md) and [Cursor Workflow](../05_AI/Cursor/Workflow.md). AB5 establishes the rule; AB6 may extend gate detail.

### Quality Expectations

- Every generation cycle links to a **brief** and **authority list**
- Revision without changed brief requires **documented reason** (review finding, not preference drift)
- Integration includes **docs updated** when behavior or design changed — same session

### Authority

| Conflict during workflow | Resolution |
|--------------------------|------------|
| AI output vs LLDL | LLDL wins — revise or reject output |
| AI output vs Gameplay | Gameplay wins |
| AI output vs domain spec (AB2–AB4) | Domain spec wins |
| Task prompt vs LLDS | LLDS wins — report conflict |
| Unclear ownership | Stop → escalate → Decisions if needed |

### Consistency Rules

- **Required reading** lists are standardized per task class — not reinvented per prompt
- Context packages favor **links to authoritative docs** over pasted prose — reduces staleness
- No skipping Research/Architecture for "quick AI generation" on major deliverables

### Future Evolution

- Workflow stages persist; **tooling adapters** change (new IDEs, APIs, agents)
- New task classes (e.g., localization pipeline) **plug into** the same spine
- Batch automation must still expose **human review checkpoints** — no silent batch ship

## 7. Collaborative Quality

**Primary responsibility:** Ensure collaborative production meets studio quality through validation, authority, and continuous improvement.

### Purpose

Collaborative Quality defines **how the studio knows output is fit** — transforming review from optional critique into governed production infrastructure.

### Participants

| Participant | Role |
|-------------|------|
| **Human — Author** | Submits work for review; responds to findings |
| **Human — Reviewer** | Engineering (Codex), Product (ChatGPT), or peer |
| **Human — Approval Authority** | Final attestation |
| **Language Model** | May assist review checklists — human records verdict |
| **QA / Test systems** | Automated validation where defined — humans interpret |

### Responsibilities

| Area | Human | AI |
|------|-------|-----|
| **Validation** | Define acceptance criteria; judge fit | Run checks humans specify |
| **Review** | Record verdicts, scores, blocking issues | Draft review notes for human edit only |
| **Authority enforcement** | Cite principles and docs | Flag potential conflicts — not resolve |
| **Continuous improvement** | Update docs when patterns fail | Suggest doc gaps — human approves changes |

### Workflow

1. **Pre-review self-check** — author verifies required reading, scope, and domain alignment
2. **Review package** (major tasks) — motivation, risk, dependencies, LLDL compliance, reviewer notes
3. **Dual review** — Codex engineering + ChatGPT product where applicable
4. **Human approval** — explicit; not implied by AI reviewer text
5. **Status hygiene** — document, index, and review package lifecycle states align — per AB1 discipline
6. **Post-ship observation** — maintenance stage feeds improvements back to docs

### Quality Expectations

| Dimension | Expectation |
|-----------|-------------|
| **LLDL compliance** | Visual and UI work extends tokens and language — no hardcoded parallel dialects |
| **Gameplay integrity** | Engine logic testable without UI; no rules in widgets |
| **Domain alignment** | Assets match AB2/AB3/AB4 system responsibilities |
| **Accessibility** | Inherited from WS9 — validated in context |
| **Documentation** | Changed behavior → changed docs + review package when major |
| **Traceability** | Review packages, version history, named approvers |

AI-generated assets pass **same philosophical gates** as handmade — briefing, LLDL, readability, accessibility, approval.

### Authority

| Truth source | Use |
|--------------|-----|
| Vision · Gameplay · LLDL | Design and product truth |
| AB0 · AB1 | Production philosophy and standards |
| AB2 · AB3 · AB4 | Domain production truth |
| AB5 | Collaborative operating truth |
| AB6 | Lifecycle gate detail — AB5 defers |
| Review packages | Handoff artifact truth for a task |

### Consistency Rules

- Review findings reference **authorities** — not vague discomfort
- Blocking issues distinguish **content** vs **lifecycle/metadata** problems
- Approved ≠ Approved — Locked; lock order preserved across Asset Bible chain
- AI must not mark its own review complete as external ChatGPT/Codex without human process

### Future Evolution

- New automated validators **plug in** — they do not replace human approval
- Quality metrics formalize in AB6 — AB5 quality philosophy remains stable
- Lessons from production failures become **doc updates** — not oral tradition

## 8. Collaborative Evolution

**Primary responsibility:** Keep collaborative production scalable and tool-independent as technology changes.

### Purpose

Collaborative Evolution ensures the operating model **survives new models, tools, and team scale** — without rewriting studio law every six months.

### Participants

| Participant | Role |
|-------------|------|
| **Human — Production Architect** | Evaluates tool changes against AB5 principles |
| **Human — Owner** | Approves material process changes |
| **Language Model / Agents** | New capabilities assessed — bounded integration |
| **Future contributors** | Inherit documented systems — not tribal tool knowledge |

### Responsibilities

| Concern | Human | AI / automation |
|---------|-------|-----------------|
| **Tool independence** | Define workflows by stage and role | Tools implement adapters |
| **Model independence** | Judge output quality against authorities | Models are interchangeable executors |
| **Process evolution** | Document changes; version AB5 when material | Suggest efficiencies — human approves |
| **Scalability** | Add roles and checkpoints before adding volume | Batch assist — not batch approve |
| **Knowledge preservation** | LLDS and Asset Bible remain source of truth | Assist migration drafts — human merges |

### Workflow

1. **Assess** — new tool/model against Collaborative Production Validation checklist
2. **Pilot** — bounded task class with full review discipline
3. **Document** — update technical docs if needed; update AB5 only if operating model changes
4. **Integrate** — roll out with required reading and role updates
5. **Audit** — periodic check that shortcuts did not skip gates

### Quality Expectations

- Technology changes must not reduce **review mandatory** status
- Pilots produce review packages like any major task
- Deprecated workflows are **archived with rationale** — per AB0 lifecycle

### Authority

- Material process changes log in [Decisions](../00_Project/Decisions.md)
- AB5 version bumps for operating model changes — not for every new model release
- AB6 may add gate metrics when evolution introduces new risk surfaces

### Consistency Rules

- **Timeless language** in AB5 — vendors named only in Related Documents, not workflow law
- Reference locks and style discipline remain **human-curated** — models do not own style memory
- Franchise-scale production reuses **systems** — per AB1 systems-over-one-offs

### Future Evolution

- Agent orchestration may automate **operator** steps — approval chain unchanged
- New media types (3D, video) **extend** domain workshops (AB2–AB4) and collaborative workflow — not bypass them
- Integration with `Asset_Bible.md` compiles AB5 into handbook form — AB5 remains authoritative workshop until superseded by locked integration doc

## 9. Conclusion

Collaborative production is one **continuous operating instrument**:

```text
Collaborative Philosophy (beliefs)
        ↓
Collaborative Roles (who)
        ↓
Collaborative Workflow (how work moves)
        ↓
Collaborative Quality (how fit is proven)
        ↓
Collaborative Evolution (how systems survive change)
        ↓
Domain production (AB2–AB4) + Lifecycle gates (AB6)
```

Each system owns one primary responsibility. Overlap creates authority confusion and unreviewed output.

### 9.2 Inheritance Summary

| Authority | AB5 inherits by… |
|-----------|------------------|
| **Vision** | Premium craft; human ownership of product intent |
| **Gameplay** | AI never redefines mechanical truth |
| **LLDL** | AI generates within visual language — does not set it |
| **AB0** | AI accelerates under direction; same review gates |
| **AB1** | Governance culture, state model, documentation discipline |
| **AB2 · AB3 · AB4** | Domain specs constrain generation and validation context |
| **AB4** | Public communication honesty extends to AI-assisted marketing assets |

### 9.3 Foundation for AB6

| Workshop | Relationship to AB5 |
|----------|---------------------|
| **AB6** | Formalizes review gates, lifecycle metrics, outsourcing handoff, and archival rules that AB5 references |

AB5 defines **how collaboration must behave**; AB6 defines **how gates are measured and recorded**.

## 10. Governance

### 10.1 Document Authority

AB5 is subordinate to Vision, Gameplay, LLDL, AB0, AB1, and domain workshops AB2–AB4 for asset meaning. AB5 is authoritative for collaborative AI-assisted production operating specifications.

### 10.2 Document Lifecycle

| State | Meaning |
|-------|---------|
| **Draft** | Active authorship |
| **Approved** | Reviewer acceptance |
| **Approved — Locked** | Human-locked; version bump for changes |

AB5 should normally be approved and locked after AB4 to preserve Asset Bible dependency order. Exceptions require explicit Human approval.

### 10.3 Review Methodology

Follow LLDS compiler workflow — review package, Codex, ChatGPT, Human approval. See [99_Reviews](../99_Reviews/README.md).

## 11. What AB5 Must Not Do

AB5 must **not**:

- Define gameplay mechanics, puzzle logic, or movement rules
- Redefine LLDL visual language or produce asset design specifications (AB2–AB4)
- Contain prompt libraries, model rankings, or vendor-specific tutorials
- Define Flutter implementation, git workflows, or CI configuration
- Define marketing strategy, budgets, or business processes
- Define formal QA metrics, outsourcing SLAs, or gate checklists (AB6)
- Duplicate AB0 §5 or AB1 governance prose — reference and extend
- Anticipate AB6 gate definitions — reference only
