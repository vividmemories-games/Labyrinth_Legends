## Task Summary

Wired shipped gameplay PNGs into the production gameplay screen: floor/wall tiles, collectibles, puzzle objects, explorer, HUD icons (back, settings, undo, erase, hint, gem, key), and `gameplay_screen.png` background. Fixed a critical `pubspec.yaml` gap — Flutter does not recurse asset folders, so gameplay sprites in `environment/`, `explorer/`, etc. were never bundled and silently fell back to `MazeTilePainter`. Added explicit subfolder entries plus an asset-manifest regression test. Layout capped board height so MOVES stats and GO CTA remain visible. Post-QA HUD polish: transparent header back/settings; enlarged icons via `LLHudIcon.visualScale` + `BoxFit.cover` (header ~49px effective, round actions 64px plate ~54px effective, stat icons 28px). In-maze objects use `Positioned.fill` at full cell with `LLGameplayAsset.visualScale` ~2.35× to compensate for PNG canvas padding (gems, keys, locks, portal, explorer). Out of scope: trap spikes, hint solver, tile autotiling/compression, victory screen pass, re-exporting HUD PNGs with transparent backgrounds.

## Motivation

0085 delivered mockup-aligned HUD chrome but maze cells still used programmatic painters because (1) code referenced PNG paths but assets were not bundled, and (2) HUD still used Material icons for some controls. On-device QA showed gold-corner painter tiles instead of stone sprites — root cause was `assets/images/gameplay/` pubspec entry not including nested category folders.

Approach: extend `LLGameplayAssets` / `LLGameplayAsset`, swap `TileView` and HUD components to image-first rendering, register each gameplay asset subfolder in `pubspec.yaml`, and add bundle tests to prevent recurrence.
