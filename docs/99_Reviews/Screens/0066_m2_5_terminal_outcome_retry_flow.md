## Task Summary

Implemented terminal outcome actions for M2.5. After execution, **lost** levels expose a **Try Again** action that calls `GameplaySession.restart()` in-place and returns to observation/planning. **Won** levels expose **Next Level**, navigating via `go_router` to the next M1-validated sequential level with a fresh `GameplaySession`. Added `LevelNavigator` for sequential progression using `LevelRepository.m1ValidatedLevelIds` — no hardcoded transitions, no unlock/progression systems. Final level (`level_003`) disables next-level with an explicit message. Engine unchanged.

**Out of scope (respected):** world map, save/load, rewards, stars, victory/failure screens, animations, sound.

## Motivation

**Why:** M2.4 exposed terminal won/lost state but left the player stuck with a disabled "Complete" button.

**Problem solved:** Players can now replay failed levels or continue sequentially through level_001 → level_002 → level_003 in one app session.

**Approach:** Retry in-place via engine `restart()`; next level via route navigation and Riverpod family provider (fresh controller + session per level).

### Retry Flow Summary

```text
Terminal Lost → Try Again
  → GameplayController.tryAgain()
  → session.restart()
  → shell phase = observation
  → retryCount++
  → board/input unlocked for replanning
```

### Next Level Flow Summary

```text
Terminal Won → Next Level
  → LevelNavigator.nextLevelIdAfter(currentId)
  → context.go('/play/$nextId')
  → GameplayScreen loads level via levelByIdProvider
  → new GameplayController + GameplaySession (Riverpod family)
  → observation/planning
```

### Level Loading Implementation

`LevelNavigator` resolves sequential IDs from `LevelRepository.m1ValidatedLevelIds` (sorted). Level definitions load through existing `levelByIdProvider` / `LevelRepository.loadWorld()` — no parallel loader.
