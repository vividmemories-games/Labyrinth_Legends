## Task Summary

Implemented a dev-only Flutter Engine Debug Sandbox that loads `level_001`, lets engineers draw a path against the real Pure Dart engine, confirm it, execute one step at a time or run to completion, inspect runtime state, and view terminal rewards. The sandbox is explicitly temporary engineering tooling and does not change engine architecture or production gameplay rules. Out of scope: production gameplay UI, persistence, progression, analytics, animations, and any M2 feature work.

## Motivation

**Why:** M1.6 completed the engine, but engineers still needed a fast way to exercise the full confirm → execute → evaluate → reward loop on device without relying on unfinished production UI.

**Problem solved:** Before this task, the engine was well-tested in Pure Dart but awkward to inspect interactively. There was no lightweight screen for loading a level, drawing paths, stepping execution, and inspecting outcome/reward state live.

**Approach:** The sandbox is a thin Flutter client over public engine APIs only. A dedicated Riverpod controller loads `LevelDefinition`, owns screen-local orchestration, and forwards calls into `GameplaySession`. Board rendering is intentionally plain and debug-first, while still using LLDL tokens/components to avoid ad-hoc styling.
